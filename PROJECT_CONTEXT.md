## Overview

Mobile detective game (React Native Expo + Node.js backend). Players solve daily mystery cases by investigating evidence, questioning suspects, reading witness statements, and making an accusation. Gamified with streaks, XP, levels, leaderboards, and a shop.

## Stack

- **Mobile**: React Native + Expo (Expo Go for dev), TypeScript, Zustand, React Query, React Navigation (bottom tabs)
- **Backend**: Node.js + Express + TypeScript, MongoDB + Mongoose, Socket.io (not yet used), JWT auth
- **Monorepo**: Yarn Workspaces — `mobile/`, `backend/`, `shared/`

## Current Status — What's Built ✅

### Backend modules (all working)

- `modules/auth` — register, login, refresh token, guest mode. JWT access (15min) + refresh (30d). **Login accepts username OR email**: the client posts `{ identifier, password }` and `auth.service.login` looks the user up via `$or: [{ email: id.toLowerCase() }, { username: id }]` (usernames are `[a-zA-Z0-9_]+` with no `@`, emails stored lowercased, so the two value-spaces never collide). `loginSchema` reads `identifier` and still falls back to a legacy `email` field so older app builds keep working. **Uniqueness**: `username` is `unique` (required), `email` is `unique`+`sparse` (optional, lowercased); register checks existence case-insensitively on email (`normalizedEmail = email.trim().toLowerCase()`) so a case-variant dup returns the friendly "Email already in use" instead of a raw index error. Mobile `LoginScreen` field is "Username or email" (`auth.identifierPlaceholder`, default keyboard, no autocapitalize/autocorrect). `auth.validator.ts` `validate<Output>()` accepts transform schemas (input `any`, returns inferred output) and throws `ValidationError` (AppError subclass) so zod failures return 422 with the field-specific message (previously threw a plain object → errorHandler fell through to generic 500 "An unexpected error occurred"). RegisterScreen also pre-validates client-side (username 3–20 chars `[a-zA-Z0-9_]`, email format, password ≥ 8) before hitting the API.
- `modules/users` — get my profile (`/users/me`), case history (`/users/me/history`), public profile (`/users/:userId/public`), push token (`POST /users/me/push-token`), **account deletion (`DELETE /users/me` → `users.service.deleteAccount`)**. Deletion is a store/GDPR requirement: it settles agency membership first (`agencies.service.removeMemberForAccountDeletion` — auto-transfers leadership to the highest-ranked/contributing remaining member, or tears the agency down if the user was the sole member), then purges every user-keyed collection (investigations, season/pass progress, event participation, achievements, challenge progress, ad rewards, daily-login, friendships both directions, profile likes given+received), then the `User` doc. Refresh tokens are stateless JWTs (no session table), so removing the user locks out lingering tokens. Mobile clears local auth via `clearAuth()` on success. Verified by a throwaway-DB cascade test (18 checks).
- `modules/cases` — daily case (`/cases/today`), recent cases with solve status (`/cases/recent`), case by id (`/cases/:id`)
- `modules/investigations` — start, get investigation, submit accusation, use hint, sync progress (`PATCH /investigations/:caseId/progress` — batch-saves inspected evidence / reviewed suspect / reviewed statement ids, no-op if `status === 'completed'`; replaces the old per-tap POST endpoints). `submitAccusation` also returns `isPerfect` (boolean — correct accusation with every clue inspected, every suspect questioned, and 0 hints used) and `redHerringStats` (`{ total, avoided, fooledBy: string[] }`). A decoy only counts as `fooledBy` when the accusation was **wrong** AND the player inspected it — a correct accusation means you saw through every red herring, so `fooledBy` is always empty on a correct/perfect run (otherwise the "all clues inspected" Perfect badge would contradict the tracker, since red herrings are themselves evidence). These are computed on submit and not persisted, so they show only on the fresh result screen, not when revisiting a completed case.
- `modules/streaks` — daily streak tracking + rewards at day 1/3/7/30/100
- `modules/rankings` — daily/weekly/all-time leaderboards + my rank
- `modules/rewards` — shop items (hints, avatars, titles), purchase, equip

### Internationalization (i18n)

- **Languages**: English (default/fallback), French (`fr`), Arabic (`ar`, RTL). Powered by `i18next` + `react-i18next`, with `expo-localization` for device-locale detection and `expo-updates` for the RTL reload.
- **Core**: `mobile/src/i18n/index.ts` — `initI18n()` (called in App.tsx before first render, gated behind `i18nReady`): resolves language from the stored `AsyncStorage["app.language"]` choice → device locale → English. `setAppLanguage(code)` persists the choice, calls `i18n.changeLanguage`, and when the writing direction flips (en/fr ↔ ar) applies `I18nManager.forceRTL` + `Updates.reloadAsync()` to re-lay-out; returns `"restart-required"` when reload is unavailable (Expo Go) so the caller shows a manual-restart alert. **On a same-direction switch (e.g. en ↔ fr) there is no reload and no refetch** — backend content arrives raw (`{en,fr,ar}`) and is resolved on render (see the content-localization bullet), so components re-render straight into the new language. `SUPPORTED_LANGUAGES` holds `{code, nativeLabel, isRTL}`; `TranslationKey` (= i18next `ParseKeys`) is the exported union used to type config objects that store keys (difficulty, tabs, statuses, etc.).
- **Typed keys**: `mobile/src/i18n/i18next.d.ts` augments i18next's `CustomTypeOptions` with `typeof en`, so `t("...")` is compile-time-checked — a typo or missing key is a tsc error. Translation files live in `mobile/src/i18n/locales/{en,fr,ar}.json` (572 keys each, kept in exact parity; `resolveJsonModule` is on).
- **Language picker**: reusable `components/ui/LanguagePicker.tsx` (three-pill row; `variant="labeled"` adds a globe + title). Used in `AppMenu` (global menu) and on the **Login + Register** screens so users can pick a language before signing in. Switching to/from Arabic reloads the app.
- **Localized content — raw + client-side resolve (instant switching)**: content is stored trilingual and the API sends it **RAW** (`{ en, fr?, ar? }` per field); the **mobile app resolves each field to the current language on render**, so switching language is instant, needs no network, and works offline. `backend/src/shared/localized.ts` — a localizable field is a plain string (legacy/English) OR `{ en, fr?, ar? }`; localizable model fields are `Schema.Types.Mixed`: **Case** (all display text + `megaOptions` + `solution.motive/weapon`), **Event** (title/description), **Season** (title/subtitle/description), **Challenge** (title/description), **Achievement** (name/description), **SeasonPass** (title/subtitle/description). **GET/read endpoints return content raw** (cases ×4, events list/get, seasons list/get/map, challenges, achievements, pass hub/rewards — no `localize*` call). **Only transient mutation results are still resolved server-side** via `localizeDeep(result, resolveLang(req))`: investigation submit/hint/reveal, event submit, season chapter start/submit (shown once, in the current language). Push notifications broadcast in English (`resolveLocalized(field, DEFAULT_LANG)`). **Mobile resolver**: `mobile/src/i18n/localizeContent.ts` — `localizeContent(value, lang)` deep-resolves any `{en,fr,ar}` object to a string (ids/numbers/usernames pass through); `useContentLang()` returns the current primary lang and re-renders on change (via `useTranslation`); `selectLocalized(lang)` is the React Query `select`. Every content query hook (`cases`/`events`/`seasons`/`challenges`/`achievements`/`pass` hooks) does `const lang = useContentLang(); useQuery({ …, select: selectLocalized(lang) })` — a language change re-runs `select` on the **cached** data (no refetch). **Answer-bearing option strings** (`megaOptions.motives/weapons`, `solution.motive/weapon`) are `{en,fr,ar}` too: mega/chapter scoring resolves the solution to `resolveLang(req)` **before** the string-compare (`norm(accusation.motive)===norm(resolveLocalized(sol.motive, lang))` in `events.service`/`seasons.service`); the client submits the chip it resolved via `useContentLang()`, and since both derive from `i18n.language` (and the axios interceptor sends `Accept-Language: getCurrentLanguage()`), the displayed option and the scored language match. Daily/mini cases grade on `suspectId` (motive is stored, not scored).
- **Content tooling**: `content-import.ts` validator accepts localized-or-string fields (helper `okText`, used for `solution.motive` too) and validates `megaOptions` for mega/chapter kinds (≥2 motives, every option non-empty, and `solution.motive/weapon` must appear among the options after resolving to English); title-keyed upsert matches string or `title.en`. `content-translate.ts` (`yarn content:translate <in.json> [out.json]`, needs `ANTHROPIC_API_KEY`) uses `@anthropic-ai/sdk` (Claude `claude-opus-4-8`, streaming) to translate an authored English case file into `{en,fr,ar}` form for `yarn content:import`. `backend/content/sample-localized-case.json` is a hand-translated demo case.
- **Launch content seed (`yarn content:launch` — NO API key needed)**: `seed-launch-content.ts` destructively wipes daily/mini/mega/chapter cases + all events/seasons + dangling participation/progress, then rebuilds a 62-day calendar (62 daily + 186 mini + 9 weekly mega events + 2 story-arc seasons × 6 chapters) **trilingual by construction**. Content is hard-coded in all three languages: the `L = { en, fr, ar }` helper (`tl(en,fr,ar)` for fixed strings, `make(l=>…)` for name/place templated glue) builds every field directly as `{en,fr,ar}` — no LLM, no translation step, deterministic and offline. The pack pools (`DAILY_PACKS`/`MINI_PACKS`/`MEGA_PACKS`/`ARC_1`/`ARC_2`) carry trilingual `place`/`role`/`alibi`/`motive`/`clue`/etc.; the `NAMES` pool pairs a Latin name (en/fr) with an Arabic transliteration; per-crime title/verb templates handle grammar per language. For mega/chapter, `solution.motive/weapon` **reference the same `L` object** as their option chip (`motiveIdx`/`weaponIdx` into `megaOptions`), so scoring resolves to the identical string in any language. The three other content seeds are trilingual too: **`seed-challenges.ts`**, **`seed-achievements.ts`** (both use an inline `tl()`), and **`seed-pass.ts`** (inline `{en,fr,ar}` objects). The old English-only **`seed-minis.ts`/`seed-events.ts`/`seed-season.ts` are deprecated stubs** that exit 1 and point to `content:launch`; `seed:all` = `content:launch && seed:pass && seed:challenges && seed:achievements`.
- **RTL layout**: the `Icon` component auto-mirrors directional icons (`back`/`forward`/`arrowRight`, via `scaleX:-1` when `I18nManager.isRTL`) so every back button and forward chevron flips app-wide from one place — never pass a custom `transform` to those names. Manual `rotate(180deg)` chevrons were replaced by the `forward` icon. Absolute-positioned badges/dots and directional padding were converted from `left`/`right`/`paddingRight` to logical `start`/`end`/`paddingEnd` (identical in LTR, auto-flipping in RTL); `flexDirection: row` and default text alignment auto-flip in RN, so most rows needed no change. `ProgressBar` fill and `GradientButton` `iconRight` flip automatically.
- **app.json**: `expo-localization` config plugin added + `extra.supportsRTL: true`.
- **Shop catalog**: shop items are a fixed backend catalog (`SHOP_ITEMS`, stable ids like `hint_x1`/`avatar_detective`); their name/description are localized **client-side by id** via `shop.items.<id>.{name,desc}` (helpers `shopItemName`/`shopItemDesc` in ShopScreen, with a fallback to the backend string if a new item ships before its translation).
- **Not translated**: brand name "Mystery Syndicate", and backend-authored *case* content (case titles/descriptions, clues, suspect names) which stays in its source language — translating authored case content is a separate content-production task.

## Key Mechanics

- **Daily case**: one case per day (`availableDate` field, format `YYYY-MM-DD`), 10 cases currently seeded covering days -3 to +6 relative to seed time
- **Investigation flow**: CaseDetail (preview, suspects names only) → Investigation (full details, 4 tabs) → Accusation modal → Result screen
- **Scoring**: `calculateScore()` in `investigations.service.ts` — base = maxScore, minus hint penalty (50/hint) and time penalty (10/min), floor at 30% if correct, flat 10% if wrong. On submit the player gains `xp = score` and `coins = floor(score / 20)`
- **Hints**: 1 per case max (`inv.hintsUsed >= 1` blocks further use). Only usable after ALL evidence inspected AND ALL suspects reviewed. Effect: randomly clears one INNOCENT suspect (marks `clearedSuspectIds`), shown with strikethrough + "✓ CLEARED" badge in the Suspects tab AND in the accusation modal, where cleared suspects are dimmed and non-selectable (can't accuse someone already proven innocent; clearing the currently-selected suspect resets the selection). Costs 1 hint from `user.hints` (default 3, buyable in shop). The charge is an **atomic conditional decrement** (`findOneAndUpdate({ hints: { $gt: 0 } }, { $inc: { hints: -1 } })`) so concurrent/double requests can never drive the balance negative; clients also clamp the displayed count with `Math.max(0, …)`.
- **Streaks**: increments daily if `lastStreakDate` was yesterday, resets to 1 if gap >1 day. Rewards at days 1/3/7/30/100 (coins + XP, defined in `streaks.service.ts` STREAK_REWARDS).
- **Levels**: progressive curve in `shared/leveling.ts` (mirrored in `mobile/src/utils/leveling.ts`) — `levelForXp = floor(sqrt(xp / 1000 + 1))`, cost to advance `n→n+1 = 3000 + (n-1)*2000` (L1→2 = 3000, L2→3 = 5000…). Denominated in thousands to match thousand-scale XP income so an avg case ≈ ½ level. Level-up triggers a modal on the result screen. **Level-up rewards** (`shared/levelRewards.ts` `levelUpRewards()`): each level grants `level * 100` coins, and every 5th level (L5/L10/L15…) is a milestone that doubles the coins **and** grants +1 free hint; `submitAccusation` `$inc`s the coins/hints and returns `levelUp = { oldLevel, newLevel, coins, hints, milestones[] }` (null if no level gained), which the InvestigationScreen level-up modal displays. The loop sums rewards across every level crossed (robust to multi-level jumps). The submit flow re-persists `user.level` from XP (self-correcting); the one-off `yarn migrate:levels` script (`scripts/recompute-levels.ts`) recomputes all stored levels at once (e.g. after a curve change) **and** clamps any negative `hints`/`coins` back to 0 (repairs accounts left negative by the old non-atomic hint decrement).
- **Weekly Mega Cases (events)** — backend `modules/events`. A weekly timed event wraps a **mega `Case`** (`Case.kind="mega"` + `eventId`; bigger cast, plus `solution.weapon`/`timelineEventId` and `megaOptions.{motives,weapons}` choice lists). `Event` model holds title/image/start+endDate/status (`upcoming→active→completed→archived`)/difficulty/`rewardConfig`/`targetCompletionSec`/`rewardsDistributed`/`notificationsSent`. `EventParticipation` (unique `(userId,eventId)` = single submission) stores the server-set clock, accusation `{suspectId,motive,weapon,timelineEventId}`, score breakdown, accuracy, rank, rewardTier. **All scoring is server-side** (`submitEvent`): suspect +500, motive +200, weapon +200, timeline +100, **no-assists +200** (requires `hintsUsed===0` AND `adHelpsUsed===0` — both hints and rewarded-ad helps forfeit it, for leaderboard fairness), speed +0–500 (`round(500·clamp((target−elapsed)/target,0,1))`); rank = score desc → time asc → finish asc. Ad-helps remain available in events (revenue) but cost the bonus like hints; the double-reward ad is daily-only and never applies to events. Exploration/hints reuse the normal `Investigation` flow on the mega case. **Rewards** (`distributeRewards`, idempotent via `rewardsDistributed`) are granted when the event ends: rank 1 title, top 10 badge, top 100 xp+coins, all participants a participation reward (titles/badges → `user.inventory`, xp/coins via the shared `grantXpAndCoins`). `events.scheduler` (node-cron every 5 min) advances statuses, distributes rewards, and fires lifecycle push notifications (soon/live/24h/last/ended, deduped via `notificationsSent`). Player API under `/api/v1/events` (list, `:id`, participate, submit, leaderboard, `:id/me`, global-leaderboard). Seed examples via `yarn seed:events` (1 active + 1 upcoming). **Mobile UI** (`features/events`): `events.service`/`events.hooks` (React Query keys `events`/`event`/`event-leaderboard`/`event-me`), `eventHelpers` (`useCountdown`, status/difficulty meta), and screens **EventListScreen / EventDetailScreen / EventLeaderboardScreen**. The **Events bottom tab** is the hub: EventList groups events into **Live Now (featured card) / Upcoming / Past**, pull-to-refresh. A **Mega Case banner** on HomeScreen shows the active/next event — tapping the card opens *that* event's detail directly, a "See all →" link jumps to the Events tab, and the banner renders nothing when there's no active/upcoming event. **Mega investigation reuses `InvestigationScreen` in event mode** (route param `eventId`): it enrolls via `participateEventApi`, the accusation modal swaps to motive/weapon/timeline pickers (from `megaOptions` + `timeline`) and submits via `submitEventApi`, showing an event result modal (score breakdown → View Leaderboard). **Pending: web admin panel** (events are seed-script-managed for now).
- **Story Arc / Seasons** — backend `modules/seasons` (retention engine; coexists with daily + mega cases). A `Season` (title/subtitle/cover/difficulty/start+endDate/totalChapters/status `upcoming→active→completed→archived`/`rewards`/`unlockCadenceDays`/`premiumEnabled` schema-only) owns N **chapter `Case`s** (`Case.kind="chapter"` + `seasonId`, `chapterNumber`, `chapterType` investigation/interrogation/discovery/twist/final_reveal, `storyText`, `cliffhanger`). **Unlock = daily + sequential**: chapter N is accessible when `now ≥ startDate + (N-1)·cadence days` AND chapter N-1 is completed (server-enforced — prevents skipping). Per-user `SeasonProgress` (unique `(userId,seasonId)`) tracks completedChapters, seasonScore, running accuracy, totalCompletionTime, chapterStats, claimedMilestones, completionClaimed, isPremium. Chapter exploration/hints reuse the normal `Investigation`. **Chapter scoring** (server-side, `submitChapter`): suspect +600, motive +200, timeline +100, no-assists +100 (zero hints AND zero ad-helps). **Rewards** (idempotent): per-chapter on completion, milestones at N chapters (`claimedMilestones`), and season-completion (title+badge+avatar via `inventory` + `grantXpAndCoins`). `cliffhanger` + `nextChapter` tease returned on submit to bait the next day. Season leaderboard ranks by score → accuracy → speed. `seasons.scheduler` (node-cron 15min) flips status, fires daily "new chapter"/"finale tomorrow"/"ending soon" pushes (deduped via `notificationsSent`). Player API `/api/v1/seasons` (list, `:id`, `:id/map`, `:id/me`, `:id/leaderboard`, `chapters/:n/start`, `chapters/:n/submit`). Seed: `yarn seed:season` ("The Shadow Syndicate", 6 chapters, back-dated for testing). **Mobile** (`features/seasons` + `features/play`): `seasons.service`/`.hooks`, **SeasonMapScreen** (Netflix-style chapter rail: completed/available/locked + progress bar + cliffhanger reveals on done chapters), **SeasonLeaderboardScreen**, surfaced via the Play tab. Chapters play through `InvestigationScreen` in **chapter mode** (route `seasonId`+`chapterNumber`): enrolls via `startChapterApi`, accusation = suspect+motive+timeline pickers, submits via `submitChapterApi`, then a **chapter result modal** shows score + the cliffhanger + next-chapter tease + rewards. **Pending: web admin + Premium Pass monetization + full 30-chapter authored content.**
- **Daily Mini Cases** — quick 1-2 min cases (`Case.kind="mini"`, `maxScore` 300). A few are seeded per day (`yarn seed:minis`), fetched via `GET /cases/mini` (`getTodayMinis`), excluded from the standalone daily/recent queries, and played through the **normal investigation flow** (daily mode). Surfaced as a horizontal **"Quick Cases"** strip on HomeScreen (tap → Investigation directly).
- **Detective of the Week** — recognition board (`GET /rankings/detective-of-week`, `getDetectiveOfTheWeek`, all-time): top accuracy (min 5 solved, via aggregation), longest current streak, most solved — top 5 each. Mobile: a **"STARS" tab** in the Rankings (`LeaderboardScreen`) rendering the three categories.
- **Friends System** — backend `modules/friends` (social foundation for future Guilds/multiplayer). `Friendship` model `{ requesterId, receiverId, status: pending/accepted/rejected/blocked }`, unique `(requesterId,receiverId)`; a block is a `blocked` row owned by the blocker. `friends.service` handles send/accept/reject/cancel/remove/block/unblock (rules: no self, no dupes, no blocked), `listFriends` (sorts: online/rank/streak/recent), `listRequests` (incoming+outgoing), `searchUsers` (username regex, paginated, excludes self+blocks), `getFriendStatus`, privacy get/update, and `touchPresence`. **Presence = `user.lastSeenAt`** (online if seen < 2 min; updated via `POST /friends/heartbeat`, called from `App.tsx` on auth + every 60s — no sockets). **Privacy** embedded on `User` (`privacy {profileVisibility public/friends/private, showOnline, showLastSeen, showStats, allowRequests}`); the public profile endpoint (`/users/:id/public`) now returns `friendStatus` + presence + **privacy-gated stats** (null + `statsHidden` when hidden). Player API `/api/v1/friends`. **Mobile** (`features/friends`): `friends.service`/`.hooks`, **FriendsScreen** (Friends / Requests / Find tabs, online dots, inline accept/add) + **PrivacySettingsScreen**, reached from the **Profile tab** (now a `ProfileStack`). `PublicProfileModal` extended with friend actions (add/cancel/accept/remove/block/unblock) and privacy-aware stats. **Pending: activity feed, friend leaderboards, comparison, Socket.IO realtime.**
- **Season Pass (monthly battle pass)** — backend `modules/pass` (long-term progression; **distinct from the Story Arc `modules/seasons`** — that's chaptered narrative, this is XP→levels→reward track). `SeasonPass` (title/banner/start+endDate/status/`totalLevels`/`xpPerLevel`/`rewards[]` with `tier` common/rare/epic/legendary + `track` free/premium/`premiumEnabled` schema-only). Per-user `PassProgress` (unique `(userId,passId)`: seasonXp, level, claimedLevels) — **season XP resets each season automatically** (new pass = new progress); claimed cosmetics persist in `inventory`. **Level = `floor(seasonXp / xpPerLevel)` capped at totalLevels** (flat curve, default 120 XP/level → 100 levels = 12k XP). **`awardSeasonXp(userId, amount, source)`** is hooked into existing flows (best-effort, never blocks): daily case +50 / mini +20 (`submitAccusation`), Mega Case +250 (`submitEvent`), Story chapter +100 (`submitChapter`). **Claiming** idempotent via `claimedLevels` (`claim` + `claim-all`); rewards applied via the shared `grantXpAndCoins` (moved to `shared/userRewards.ts` to break the pass↔investigations cycle) + `inventory`. Leaderboard by seasonXp (global + **friends** via the friend graph). `pass.scheduler` (node-cron 15min) handles status + started/final-week/last-day pushes. API `/api/v1/pass` (hub `/`, `/rewards`, `/claim`, `/claim-all`, `/leaderboard`). Seed: `yarn seed:pass` ("The Crimson Conspiracy", 100 levels, formula-driven reward map). **Mobile** (`features/pass`): `SeasonPassHubScreen` (banner, level, progress, days left, unclaimed) + `PassRewardsScreen` (track with per-level claim + Claim All), surfaced as a **Season Pass card on the Play hub**. **Pending: Challenges, Collectibles, season summary, Premium track wiring.**
- **Seasonal Challenges** — backend `modules/challenges` (daily/weekly/monthly objectives that grant Season-Pass XP). `Challenge` definitions (`key`, `period`, `metric`, `target`, `rewardSeasonXp`/`rewardCoins`, recur every period) + per-user `ChallengeProgress` (unique `(userId,challengeKey,periodKey)`; periodKey = `YYYY-MM-DD`/`YYYY-Www`/`YYYY-MM`). **Counter-based** via `recordChallengeEvent(userId, metric, amount)`, hooked best-effort into the submit flows: `case_solved` (all), `mini_solved`, `no_hint_solve`, `mega_completed`, `chapter_completed`. Completing is automatic; **claiming** grants the reward (`awardSeasonXp` + `grantXpAndCoins`), idempotent via `claimed`, with claim-all. Import direction: challenges→pass→shared (no cycle). API `/api/v1/challenges` (list / claim / claim-all). Seed: `yarn seed:challenges` (upsert by key). **Mobile** (`features/challenges`): `ChallengeCenterScreen` grouped Daily/Weekly/Monthly with progress bars + claim, reached from the **Season Pass Hub** "Challenges" row. **Pending threshold metrics (accuracy/season-level/streak goals) — only counter metrics so far.**
- **Achievements** — backend `modules/achievements` (Xbox/Steam-style prestige layer; **auto-unlock**). `Achievement` definitions (`key`, `category` cases/accuracy/streaks/story/mega/social/seasonal, `rarity` common/rare/epic/legendary/mythic → `RARITY_POINTS` 10/25/50/100/250, `metric`, `progressType` counter|value, `target`, optional reward + `hidden`) + per-user `UserAchievement` (unique `(userId,achievementKey)`; snapshots category/rarity/points for score/leaderboard without joins). **`recordAchievementEvent(userId, metric, amount)`** — counter (`+=`) or value (`max`) — hooked into the submit flows + streaks + friends: `case_solved`, `perfect_case`, `mega_completed`, `chapter_completed`, `streak_day` (value), `friend_added`. On hitting target → **auto-unlock**: grant points + reward (`grantXpAndCoins` + inventory) + push, no claim. **Achievement Score** = Σ points of unlocked (also on `/users/me` + public profile). API `/api/v1/achievements` (list grouped + `/leaderboard`). Seed: `yarn seed:achievements` (~19, upsert). **Mobile** (`features/achievements`): `AchievementsScreen` (score card + category sections + rarity-colored progress/unlock cards), reached from a **trophy icon on the Profile header**. **Note:** secondary progression hooks (pass/challenge/achievement) are now **awaited** (`Promise.allSettled`) in the submit flows so refetch-after-submit sees fresh values (a race the `yarn smoke` test caught). **Pending: badge/title galleries, showcase, trophies, evidence book, collectibles, feed, threshold/seasonal metrics.**
- **Detective Agencies (Guilds)** — backend `modules/agencies` (core social/competition pillar). `Agency` (unique `nameLower`, privacy `public`/`request`, leaderId, memberCount, `agencyXp`/`level` via `agencyLevelForXp` loop curve [L1 1000, L2 3000, L3 6000…], `memberCapForLevel` = 20+5·lvl capped 100, `weeklyPoints`/`seasonalPoints`, `lastWeekKey`). `AgencyMember` (**unique `userId`** = one agency per user; role leader/coleader/officer/member via `ROLE_RANK`, contributionTotal/weeklyContribution) + `AgencyJoinRequest` (unique `(agencyId,userId)`). `agencies.service`: create (name validation + offensive filter), list/search, join (public instant / request → approval), leave (leader must transfer or delete), requests approve/reject (officer+), role promote/demote (coleader+, below own rank), kick (officer+, below own rank), transfer leadership, delete (leader), `getMyAgency`/`getAgencyView` (members + presence via friends `isOnline`), weekly/global leaderboard. **`addAgencyContribution(userId, points, source)`** hooked into the submit flows (case +10, mini +5, mega +100, chapter +20) → member + agency XP/weekly/seasonal; `agencies.scheduler` (hourly) resets `weeklyPoints`/`weeklyContribution` on a new ISO week. Points are **server-side only** (anti-cheat). API `/api/v1/agencies`. **Mobile** (`features/agencies`): `AgenciesScreen` (branches: dashboard + members + moderation via AppPopup + join requests when in an agency; else browse/search + Create), `CreateAgencyScreen`, `AgencyLeaderboardScreen` (weekly/global) — reached from a **scales icon on the Profile header**. **Pending: agency chat (real-time + moderation), store, events, agency seasons/achievements/showcase, invites, regional boards.**
- **Shop items** (`rewards.service.ts` SHOP_ITEMS): hint_x1/x3/x5 (150/400/600 coins — priced premium so hints are a real choice and the planned watch-an-ad-for-a-hint reward stays attractive). **6 avatars** (cheap→prestige): detective 200, masked 350, inspector 700, prodigy 1000, ace 1400, legend 2000. **6 titles**: shadow 400, arbiter 600, mastermind 1200, maverick 1600, decorated 2200, phantom 3000. Avatars/titles are one-time purchases added to `user.inventory`, then "equipped" via separate endpoint (sets `user.avatar` or `user.title`). **Level-gated top tier**: items can carry `requiredLevel` (avatar_legend = L10, title_phantom = L15); `purchaseItem` rejects if `levelForXp(user.xp)` is below it, and ShopScreen shows a "Requires Level X" lock instead of the price. ## Testing

- **`yarn smoke`** (backend, `scripts/smoke.ts`) — dependency-free smoke test. Spins up a throwaway DB (`mystery-syndicate-smoketest`, dropped at the end), creates a user + case + active pass + challenge, runs `submitAccusation`, and asserts the full progression fan-out: account XP/coins, Season-Pass XP + level, challenge progress, challenge claim → Season XP, pass-level claim → coins, claim idempotency, and locked-level guard. Exits non-zero on any failure (CI-friendly). Run it after touching the case-submit hot path or any cross-cutting hook (`grantXpAndCoins`, `awardSeasonXp`, `recordChallengeEvent`).

## Known env / setup details

- Backend runs on port 4000, MongoDB Atlas (had to add `tls: true, tlsAllowInvalidCertificates: true` to mongoose.connect due to Node 22 + Atlas SSL issue)
- Mobile `.env`: `EXPO_PUBLIC_API_URL=http://<LAN_IP>:4000/api/v1` — must match dev machine's local network IP for Expo Go to reach the backend
- AsyncStorage used instead of expo-secure-store (SecureStore had native module issues in Expo Go)
- JWT expiry errors (`TokenExpiredError`) were returning 500 — fixed in `errorHandler.ts` to return 401, which triggers the Axios refresh-token interceptor in `mobile/src/services/api.ts`
- React Query cache is cleared on logout via `queryClient.clear()` (imported from `mobile/src/services/queryClient.ts`)

## Not Yet Built (roadmap)

1. **Notifications Push** (Expo Push Service) — **code complete, gated on EAS build + credentials.** Architecture: `expo-notifications` on device → Expo push token sent to backend (`POST /users/me/push-token`, stored on `user.pushToken`) → backend pushes via `expo-server-sdk`. Backend `modules/notifications` has `notifications.service.ts` (`sendToTokens`, `notifyNewCase`, `notifyStreakReminder`; clears tokens reported `DeviceNotRegistered`) and `scheduler.ts` (`node-cron`: 09:00 new-case, 20:00 streak reminder; timezone via `CRON_TZ` env), started from `server.ts` (skipped when `NODE_ENV=test`). Mobile: `src/services/notifications.ts` (`registerForPushNotificationsAsync` + `syncPushToken`, called from `App.tsx` on auth; fails soft in Expo Go), `expo-notifications` plugin + `extra.eas.projectId` in `app.json`. **Remaining (not code):** `npx expo install expo-notifications expo-constants` in mobile, `eas init` (fills `projectId`), Android FCM v1 + iOS APNs credentials via EAS, and a dev build (remote push does NOT work in Expo Go on SDK 53+).
2. **Automatic case generation** via Claude API — daily scheduler generates a new case via LLM so content doesn't run out. Requires Anthropic API credits (not yet purchased).
3. **EAS Build** — needed to publish to Play Store; required for push notifications and AdMob to work (Expo Go doesn't support them)
4. **Ads (AdMob rewarded, SSV-secured)** — **code complete, gated on a rebuild + AdMob account + public backend URL.** Opt-in rewarded ads only (no banners/interstitials) via `react-native-google-mobile-ads`. Mobile `src/services/ads.ts` (`initializeAds` from `App.tsx`; `showRewardedAd({ userId, customData })` sets `serverSideVerificationOptions`, resolves true only on earned reward; lazy-requires the native module so it no-ops in Expo Go / pre-ads builds; uses Google TEST ad units in `__DEV__`). Config plugin + test App IDs in `app.json`. **Rewards are granted ONLY via AdMob server-side verification (SSV) — there is no client-callable grant endpoint.** Flow: client shows ad with `userId` = user id and `customData` = `{caseId,type}` → AdMob calls `GET /api/v1/ads/ssv` (public, in `modules/ads`) → `ads.service` verifies the ECDSA signature against Google's verifier keys (`gstatic.com/admob/reward/verifier-keys.json`, cached), dedupes by `transaction_id` (`ProcessedAdReward` model), then calls the grant service. Reward `type`s: `eliminate` (clear a suspect), `reveal` (expose a red herring) — both count toward `inv.adHelpsUsed` (disqualifies the Perfect badge) — and `double` (`claimDoubleReward`, doubles case XP+coins via shared `grantXpAndCoins`, can trigger a level-up, guarded by `inv.rewardDoubled`). The grant services (`adEliminateSuspect`/`adRevealRedHerring`/`claimDoubleReward`) live in `investigations.service` but are invoked only by the SSV handler. Client reconciles asynchronously: after the ad earns, `InvestigationScreen` polls `getInvestigationApi` until the server-applied change appears, then updates the store/UI. **Remaining (not code):** create AdMob account + real rewarded ad unit IDs (swap into `ads.ts` `REWARDED_AD_UNIT_ID`, app IDs in `app.json`), set each ad unit's **SSV callback URL** to `https://<public-backend>/api/v1/ads/ssv` in the AdMob console, deploy the backend publicly (SSV can't reach localhost/LAN — use a tunnel like ngrok to test), and **a new `eas build`** (native module added).
